The Concept

📐 What is a Blueprint?

A Blueprint is a stencil for your content. It tells Statamic what fields to show editors in the control panel, and it defines the data schema your Antlers templates consume on the front end.

No Blueprint = no fields. It's that central.

Simple idea

Blueprint = what fields your content has, and how they're arranged for editors

Ownership

🗂️ What Does a Blueprint Belong To?

Almost every content area in Statamic has a blueprint attached to it — here's the breakdown:

AreaBlueprint rules
Collections Can have multiple — each entry picks one
Taxonomies Same — each term can pick a blueprint
Global Sets One blueprint each
Asset Containers One blueprint each
Forms One blueprint each
Users All users share a single blueprint
Tip

Collections and taxonomies support multiple blueprints — great for different post types (e.g. basic_post vs art_directed_post).

Getting Started

🛠️ Creating a Blueprint

Three ways — pick your style:

🖱️
Control Panel → Collection / Taxonomy area

Each content area has a built-in blueprint editor. Click in, drag fields around, done.

🗂️
Control Panel → Blueprints hub

A central page to jump between blueprints across all areas at once.

✍️
Drop a YAML file in resources/blueprints/

The developer way. Write YAML, Statamic picks it up instantly. No UI needed.

File Layout

📁 Directory Structure

All blueprints live as YAML files under resources/blueprints/. The folder structure mirrors your content areas.

Collections & taxonomies → subdirectory named after the handle
Globals, Assets, Forms → one YAML file per item handle
The Schema

🧬 YAML Structure

A blueprint is just sections → fields. Sections become tabs in the publish form when you have more than one.

basic_post.yaml 
sections:
  main:
    display: Main
    fields:
      -
        handle: content
        type: markdown
  meta:
    display: SEO Metadata   # becomes a tab
    fields:
      -
        handle: meta_title
        type: text
      -
        handle: meta_description
        type: textarea
Hot Tip

Fields are indexed as an array, not keyed by handle. This lets you reference the same field multiple times and override settings inline — very handy for Bard, Replicator, and Grid sets.

Field Definition

🔧 Defining Fields

Fields are the heart of a blueprint. Two flavours: inline (defined right there) or references (pulled from a fieldset).

Inline field
fields:
  -
    handle: title
    field:
      type: text
      required: true
Field reference
fields:
  -
    handle: content
    field: common.content
               ↑ fieldset.field_handle
Override

Override a referenced field's config with a config key. Any keys there win over the fieldset definition.

DRY Fields

♻️ Reusable Fields via Fieldsets

Pre-configure common fields in a fieldset and pull them into any blueprint. Update once → reflects everywhere.

referencing a fieldset field 
fields:
  -
    handle: hero_content
    field: common.content  # reuse!
  -
    handle: body_content
    field: common.content  # use it again
    config:
      display: Body Copy       # override label only
      validate: required
Same field used multiple times with different handles
Override only what you need — everything else inherits
Fieldset Import

📦 Importing Entire Fieldsets

Pull an entire fieldset into a blueprint with the import key. Use a prefix to namespace the fields — and to safely import the same fieldset more than once.

The fieldset (survey.yaml)
fields:
  -
    handle: food
    type: text
  -
    handle: food_reason
    type: textarea
Import it twice in a blueprint
fields:
  -
    import: survey
    prefix: favorite_
  -
    import: survey
    prefix: least_favorite_
Result

You end up with favorite_food, favorite_food_reason, least_favorite_food, and least_favorite_food_reason — all from one fieldset.

Watch out

Skip the prefix and you can only import the fieldset once — handles would collide and overwrite each other.

Smart Fields

👁️ Conditional Fields

Show or hide fields based on the value of other fields. Build smarter publish forms without overwhelming editors.

🖼️
Show caption only if an image is selected

If the asset field is empty, the caption field vanishes — no clutter.

🔀
Show a block of fields when a toggle is on

Flip a "Show sidebar?" toggle → sidebar fields appear. Clean.

Docs

Conditional logic has its own dedicated documentation page at statamic.dev/conditional-fields — the rule builder in the control panel makes it point-and-click.

Data Integrity

✅ Validation

Fields can enforce rules before saving. Statamic uses Laravel's built-in validation rules plus some Statamic-specific extras for Grids, Bards, and conditional logic.

validation rules in a field 
-
  handle: title
  field: common.title
  config:
    validate: required|max:200  # Laravel rules
Where

In the control panel, switch to the Validation tab while configuring a field to use the rule-picker UI instead of writing YAML by hand.

Repeatable Rows

🔢 Grid Fieldtype

The Grid field lets editors repeat a set of sub-fields as many times as they like — perfect for link lists, team members, pricing rows, and more.

grid field definition 
links:
  type: grid
  fields:
    -
      handle: url
      field: links.url       # reference from fieldset
    -
      handle: text
      field: links.text
    -
      handle: external
      field: links.external
Pro

Grid fields use the same blueprint field syntax — so you can reference fieldset fields and import fieldsets inside a grid too.

Entry Listings

🙈 Unlisted Fields

Control whether a field appears as a column in the entry listing table — separate from conditional field visibility in the publish form.

Hide column completely
listable: false
# Never shown in listings
Hidden by default, togglable
listable: hidden
# Off by default, user can re-enable
Use case

Toggle fields and long text fields rarely make sense as listing columns — set them to listable: false to keep the table clean.

Best Practices

💡 Pro Tips

🧩
Start with one section, split later

A single-section blueprint is just a flat list of fields. Only add a second section (tab) when the form gets long enough to need it.

📁
Store shared fields in fieldsets

Any field you use in 2+ blueprints belongs in a fieldset. Change it once, done everywhere — the DRY dream.

🏷️
Keep handles snake_case and descriptive

hero_image not img1. Your Antlers templates will thank you six months from now.

👁️
Use conditional fields aggressively

A simpler-looking form for editors = fewer mistakes. Hide fields that aren't relevant until they're actually needed.
