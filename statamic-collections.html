Introduction

📦 What is a Collection?

A Collection in Statamic is a structured container used to organize related content. Think of it as the foundation of your content architecture.

📁
A Folder

Physically lives as a directory inside your project.

🧠
A Database Table

Acts like a table, but file-based — no DB required.

📝
A Content Type

"Blog", "Products", "Docs" — each is a collection.

In simple terms

Collection = Rules + Structure + Content Group

Architecture

🧠 Concept Overview

A collection defines three essential things about your content:

01
How content is structured

Defined via blueprints — field schemas for your entries.

02
How URLs are generated

Controlled via routes — flexible pattern-based URL building.

03
Default behaviours

Publishing rules, sort direction, injected default values, and more.

On Disk

📁 File Structure

Collections live inside your project under content/collections/. Each collection is a folder containing its entries and a config file.

project structure
content/
└── collections/
    └── blog/
        ├── first-post.md
        ├── second-post.md
        └── blog.yaml
.md

Files are entries — your actual content, written in Markdown.

.yaml

The collection config — routes, templates, sort order, defaults.

YAML

⚙️ Collection Configuration

Each collection has a .yaml config file that controls its core behaviour:

blog.yaml
title: Blog
route: /blog/{slug}
template: blog/show
sort_dir: desc
route

Defines the URL structure for every entry in this collection.

template

Controls which Antlers/Blade template is used for frontend rendering.

Content Units

📝 Entries

Entries are the actual content items inside a collection. Each .md file is one entry, containing frontmatter (YAML) and body content (Markdown).

first-post.md
---
title: Tony Hawk lands the first-ever 900
id: 3a28f050
---

Tony Hawk made history by landing the 900 at the 1999 X Games.
Title
Required. Display name of the entry.
Content
Markdown body below the frontmatter.
Metadata
YAML frontmatter fields.

👉 What you write in the UI is saved as Markdown behind the scenes.

Field Definitions

🧩 Blueprints

Blueprints define the structure of your entries — which fields exist, what type they are, and how they appear in the Control Panel.

Default Fields
title content slug author date
➕ Add Custom Fields

Text, images, booleans, selects, relationships — any field type you need.

📋 Multiple Blueprints

A single collection can have multiple blueprints — useful when entry types differ.

Use Case

A "Reviews" collection might have separate blueprints for "Book Reviews" and "Movie Reviews" — each with different custom fields.

Identifiers

🎯 Titles

Every entry requires a title. You can also automate it using a format string — perfect for consistent naming patterns.

collection.yaml
title_format: "{stars} star rating by {author:name}"
✨
Example output
5 star rating by John Doe
URL Identifiers

🔗 Slugs

Slugs are URL-friendly versions of titles, automatically generated and used in your routes.

Title My First Blog
→
Slug my-first-blog
→
URL /blog/my-first-blog
Disable slugs
# If slugs are disabled, entry ID is used as filename
slugs: false
Scheduling

📅 Dates & Visibility

Collections can use dates to automatically control entry visibility — no manual publishing needed.

collection.yaml
date: true
date_behavior:
  future: private
  past: public
public

Visible everywhere and in all listings.

unlisted

Hidden from listings but direct URL works.

private

Completely hidden. Returns 404.

Computed State

📊 Entry Status System

Each entry gets a computed status based on its date and publish settings. Use status (not published) for filtering.

Published
Live and visible to all users
published
Scheduled
Future-dated, not yet live
scheduled
Expired
Past the expiry date
expired
Draft
Manually hidden from public
draft
Time Support

Enable time alongside dates to publish at exact times (e.g. 10:30 AM) and maintain correct order for same-day entries.

Structure

📚 Ordering & Structure

Enable manual drag-and-drop ordering for custom content hierarchies.

collection.yaml
orderable: true
⠿
Drag & Drop UI

Reorder entries visually in the Control Panel.

🌳
Custom Hierarchy

Build nested content trees for docs and structured sites.

Hierarchical Content

🌳 Nested Content (Structure)

Collections can be hierarchical — great for documentation, category trees, or nested pages.

structure example
📁 Docs
 ├── Getting Started
 ├── API
 │   └── Authentication
 └── Deployment
Limit nesting depth
max_depth: 1
Root Page

Define a root entry that acts as / . Useful for homepages and section landing pages.

URL System

🌐 Routing

Routes define how URLs are generated for entries using dynamic placeholders wrapped in {curly braces}.

Blog style
route: /news/{year}/{month}/{day}/{slug}
SEO-focused
route: /{parent_uri}/{slug}.html
Minimal (ID-based)
route: /{id}
Pro Tip

Plan your routes early to avoid restructuring later — changing routes breaks existing URLs and can hurt SEO.

Navigation

🔁 Redirects

Entries can act as redirects — perfect for external links or navigation shortcuts.

Simple redirect
redirect: https://example.com
With status code
redirect:
  url: https://example.com
  status: 301
Categories & Tags

🏷️ Taxonomies

Define taxonomies at the collection level — they'll be automatically available in all entries.

collection.yaml
taxonomies:
  - categories
  - tags
Page Attachment

🔗 Mounting Collections

Attach a collection under a specific page to automatically nest its URLs beneath that page's path.

collection.yaml
mount: blog-page-id
Result:
/blog/post-1
/blog/post-2

Best Practices

💡 Developer Tips

⚡
Use Control Panel for Speed

The CP is optimised for fast content creation — great for day-to-day editing without touching files.

🔧
Code Editor for Precision

Prefer your code editor + git for version-controlled, team-friendly content workflows.

🚧
Plan Routes Early

Restructuring routes later breaks URLs. Design your URL schema before you start publishing content.

🧩
Keep Blueprints Reusable

Design clean, modular blueprints. Avoid duplicating field definitions — reuse across collections where possible.
