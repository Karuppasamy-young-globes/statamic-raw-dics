The Concept

🔲 What is a Field?

A field is a single input in the control panel. It has a type that determines what it looks like and how it stores data, and a handle that becomes the variable name in your templates.

Fields live inside Blueprints (which attach to content areas) or Fieldsets (which are reusable libraries). Every field shares the same core config settings regardless of its type.

Simple idea

Field = one input with a type, a handle, and a bunch of shared settings

Configuration

⚙️ Common Settings

Every field — regardless of type — shares these settings. Configure them in the control panel's field editor or directly in YAML.

SettingWhat it does
display The label shown to editors in the control panel
handle The variable name used in Antlers templates — e.g. {{ title }}
instructions Help text displayed below the field to guide authors
listable Whether to show this field as a column in the CP listing table
sortable Whether editors can sort the listing table by this field
visibility Controls how the field appears on publish forms (see below)
always_save Overrides data flow — forces value to be submitted even when hidden
localizable Whether this field can have different values per site in a multi-site setup
a fully-configured field 
-
  handle: hero_title
  field:
    type: text
    display: Hero Title
    instructions: Keep it punchy — max 60 chars
    validate: required|max:60
    listable: true
    localizable: true
Display Control

👁️ Field Visibility

Every field is visible by default. Use the visibility setting to control exactly how it behaves in the publish form — four modes to choose from.

Visible
Always shown, always editable

The default. Editors see the field, they can change it, the value is saved on submit.

Read Only
Shown, but not editable

Editors can see the current value but can't change it. The value is still saved on submit.

Computed
Read-only computed display — value is never saved

Shows a dynamically calculated value for reference. Nothing is submitted on save — the value always comes from the compute logic.

Hidden
Not shown — but current value is saved

The field is invisible in the publish form, but whatever value it already has will be carried forward on save. Great for internal/system fields.

Hot Tip

You can also dynamically show and hide fields using Conditional Fields and Revealer Fields — great for form UX that adapts to what the editor picks.

Note

Hiding a field via conditions (not static visibility) will generally prevent its value from being submitted on save. See Field Data Flow below for the full picture.

Save Behaviour

🌊 Field Data Flow

By default, every field submits its value on save. There are only two situations where a field's value is not submitted:

Field visibility is set to Computed
Field is dynamically hidden by a field condition

Need to override the condition behaviour? Two options:

🔓
Use a Revealer Field

A Revealer is a special fieldtype that toggles the visibility of surrounding fields while always allowing their values to be saved — no data loss.

💾
Set always_save: true

Forces the field's current value to be included in the save payload regardless of visibility — even if hidden by a condition.

Where Fields Live

🗂️ Blueprints & Fieldsets

Fields always live inside one of two containers. The choice depends on whether you need them in one place or many.

📐 Blueprint

Attached to collections, taxonomies, globals, assets, forms, and users. Controls the publish form layout — field order, width, sections (tabs).

Define inline for one-off fields
Reference fieldset fields for reuse
🧩 Fieldset

Not attached to anything — a library of pre-configured fields. Import individual fields or the whole fieldset into blueprints.

DRY: update once, reflects everywhere
Also works inside Grid, Replicator & Bard
Quick rule

Used in one blueprint? Define it inline. Used in two or more? Move it to a fieldset. Future-you will be grateful.

Best Practices

💡 Pro Tips

🏷️
Handle = your template variable — name it well

hero_title not ht. featured_image not img. You'll be typing these in Antlers for years.

✍️
Write good instructions

The instructions field is prime real estate. Use it — "Keep under 60 chars for SEO" beats "Title" alone every time.

👁️
Use Hidden visibility for system fields

Fields like internal IDs, computed slugs, or migration flags don't need to be visible. visibility: hidden keeps them out of the way while still saving the value.

🌍
Set localizable: true from the start

If there's any chance the project goes multi-site, flip this on now. Retrofitting localisation onto existing content is painful.
