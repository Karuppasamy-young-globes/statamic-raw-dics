The Concept

🧩 What is a Fieldset?

A Fieldset is a named collection of pre-configured fields that lives on its own — not attached to any content area. Think of it as your field library.

You build a fieldset once, then pull its fields into as many blueprints as you like. Update the fieldset → every blueprint that references it gets the change instantly.

Simple idea

Fieldset = a reusable field library that blueprints borrow from

Know the Difference

⚖️ Fieldsets vs Blueprints

They look similar in YAML but serve very different roles.

📐 Blueprint
Attached to collections, forms, etc.
Has sections (= tabs in publish form)
Renders the control panel publish UI
Defines your front-end data schema
🧩 Fieldset
Not attached to anything directly
No sections — fields only
Imported into blueprints (or fieldtypes)
Single source of truth for shared fields
Rule of thumb

If a field appears in 2+ blueprints → it belongs in a fieldset. If it's unique to one blueprint → define it inline.

Usage Scope

🗺️ Where Can You Use Fieldsets?

Fieldsets aren't limited to blueprints — they plug into complex fieldtypes too, making them doubly useful.

📐
Inside Blueprints

Reference individual fields or import the whole fieldset with a prefix. The most common use case.

🔢
Inside Grid fields

Define a row's sub-fields using a fieldset — keeps your Grid definition clean and reusable across multiple Grids.

🧱
Inside Replicator & Bard

Each Replicator set or Bard block can pull its fields from a fieldset — e.g. bard-image.yaml, bard-quote.yaml.

Getting Started

🛠️ Creating a Fieldset

Two ways — pick your style:

🖱️
Control Panel → Fieldsets

Navigate to the Fieldsets area, click New Fieldset, drag and drop fields in the builder. YAML gets written for you.

✍️
Drop a YAML file in resources/fieldsets/

The developer way. Write YAML, save, done. Statamic picks it up immediately — no restart required.

File Layout

📁 Directory Structure

All fieldsets live as YAML files under resources/fieldsets/. You can nest them in subdirectories — just use . in the handle to reference them.

resources/fieldsets/
resources/
 └── fieldsets/
    ├── common.yaml← handle: common
    ├── seo.yaml← handle: seo
    ├── bard-image.yaml← handle: bard-image
    ├── bard-quote.yaml
    └── blocks/← subdirectory
        └── hero.yaml← handle: blocks.hero
Subdirs

A fieldset at fieldsets/blocks/hero.yaml is referenced as blocks.hero — dots map to directory separators.

The Schema

🧬 YAML Structure

Dead simple — just a flat list of fields. No sections, no nesting. That's the whole file.

common.yaml 
fields:
  -
    handle: content
    field:
      type: markdown
      display: Content
  -
    handle: featured
    field:
      type: toggle
      display: Featured?
      default: false
Fields are fully configured here — type, display name, validation, instructions, everything
No sections: key — fieldsets are flat by design
Putting It to Work

🔗 Using Fieldsets in Blueprints

Fieldsets don't do anything on their own — they come alive when referenced from a blueprint. Two approaches:

① Reference a single field
blueprint.yaml — single field ref 
fields:
  -
    handle: body
    field: common.content   # fieldset_handle.field_handle
  -
    handle: is_featured
    field: common.featured   # same fieldset, different field
    config:
      display: Mark as Featured  # override just the label
OR
② Import the whole fieldset
blueprint.yaml — full import with prefix 
fields:
  -
    import: seo              # pulls in all fields from seo.yaml
    prefix: seo_            # prefixes every field handle
  -
    import: seo              # import again for another context
    prefix: og_             # different prefix = no handle collision
Watch out

Always use a prefix when importing a fieldset more than once in the same blueprint — without it, field handles collide and overwrite each other.

Advanced Usage

🧱 Inside Grid, Replicator & Bard

Fieldsets really shine when you need consistent sub-fields inside complex fieldtypes. Define once, use everywhere.

Bard set using a fieldset 
# In your blueprint's bard field config:
content:
  type: bard
  sets:
    image_block:
      display: Image
      fields:
        -
          import: bard-image    # pulls fields from bard-image.yaml
    quote_block:
      display: Quote
      fields:
        -
          import: bard-quote    # each block type = its own fieldset

Best Practices

💡 Pro Tips

📄
Create a common.yaml first

Most projects have 3–5 fields that appear everywhere — title variants, SEO fields, featured toggles. Get them into a fieldset on day one.

🗂️
Organise by context, not by field type

seo.yaml, social.yaml, hero.yaml — group by where they're used, not by whether they're text or toggle fields.

🧱
One fieldset per Bard/Replicator set type

Give each block its own file: bard-image.yaml, bard-video.yaml. Easier to find, easier to edit, and they compose cleanly.

🏷️
Always use prefix on import

Even if you only import once — adding a prefix makes your data keys self-documenting: seo_title is clearer than just title.
