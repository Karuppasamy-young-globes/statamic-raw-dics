Introduction

🌐 What are Globals?

Global variables store content that belongs to the whole site, not just a single page or URL. Globals are available everywhere — in all of your views, all of the time.

🌍
Always Available

Accessible from every template, partial, and layout without passing data manually.

🗂️
Flat Files

Stored as simple YAML files — no database, fully version-controllable.

🧩
Organised in Sets

Grouped into named sets for structure and scoped template access.

In simple terms

Globals = Site-Wide + Always Accessible + Reusable Content

When to Use

🎯 What Globals Are For

Globals are intended for reusable content or content that belongs to the site rather than a single page. Common examples include:

01
Company Information

Phone number, address, logo, social media handles — things that appear across many pages.

02
Footer & Navigation Content

Footer text, copyright notices, nav labels — shared layout content that editors control without touching code.

03
Testimonials & Quotes

Customer quotes, pull quotes, or featured reviews that rotate across the site.

04
Site Settings & Toggles

On/off feature flags, maintenance mode banners, cookie consent copy — operational knobs for editors.

05
Success & Error Messages

Form feedback text, 404 page copy, validation messages — centralised and editable without deployments.

Organisation

📦 Global Sets

Globals are organised into sets, each containing a group of related fields. This keeps things tidy and each set also acts as a scope for templating purposes.

🗃️
Grouping

Keep related globals together — footer, company, settings — rather than one giant flat list.

🔭
Scoping

Each set name becomes a template prefix — footer:copyright, company:phone.

On Disk

💾 Storage

Globals are stored in the content/globals directory. Each global set has two YAML files — a metadata file and a data file.

project structure
content/
└── globals/
    ├── global.yaml
    ├── footer.yaml
    └── default/
        ├── global.yaml
        └── footer.yaml
{handle}.yaml

The metadata file — contains the set title and field configuration.

{site}/{handle}.yaml

The data file — stores the actual values per site.

Note

The content/globals/{handle}.yaml file holds metadata like the set title, while actual field data is in content/globals/{site}/{handle}.yaml.

Configuration

📄 YAML File Examples

Here's how the two YAML files relate to each other for a footer global set:

# Metadata — content/globals/footer.yaml

footer.yaml
# content/globals/footer.yaml

title: Footer

# Data — content/globals/default/footer.yaml

default/footer.yaml
# content/globals/default/footer.yaml

copyright: 2021 Neat Fake Company, LLC
flair: Made with ❤️ by humans
Frontend

🖥️ Frontend Templating

All variables inside a global set are accessed through a scoped prefix matching the set handle — for example, footer:copyright.

footer.antlers.html
<footer class="site-footer">
  <p>{{ < /span > < span class="text-white" class="text-acid" }}</p>
  <p class="text-sm">{{ < /span > < span class="text-white" class="text-acid" }}</p>
</footer>
Tip

In this example, all variables inside the footer global set are accessed via footer:<var_name>. The set handle becomes the scope prefix in your templates.

Access Patterns

🔭 Scoping & Variable Access

If you only have one default global set (typically named "Globals"), the scope prefix is optional. You can access variables two ways:

Without Scope
{{ < /span > < span class="text-white" class="text-acid" }}

Works for the default "global" set only

With Scope
{{ < /span > < span class="text-gem-300" class="text-white" class="text-acid" }}

Explicit — always works, always clear

{{ footer:copyright }}

Scoped access — reaches into the footer set for the copyright field.

{{ global:site_name }}

Explicit scope on the default set — equivalent to {{ site_name }}.

{{ site_name }}

Unscoped — works only for the default global set (named "global").

Best Practice

Always use the explicit scope (global:var_name) even for the default set — it avoids variable naming conflicts and makes templates easier to understand at a glance.

Reference

🧠 Quick Recap

Concept Meaning
Global Site-wide variable accessible in every view and template
Global Set A named group of related global fields (e.g. "footer", "settings")
Handle The machine-readable identifier for a set — used as the template scope prefix
Scope The set handle prefix used in templates (footer:copyright)
Metadata file content/globals/{handle}.yaml — stores title and blueprint info
Data file content/globals/{site}/{handle}.yaml — stores actual field values
Default set The "global" set — its scope prefix is optional in templates
Best Practices

💡 Developer Tips

🗃️
Group Logically

Use separate sets for distinct concerns — don't throw everything in one "global" set. company, footer, settings each deserve their own set.

🔭
Always Use Explicit Scope

Prefer {{ global:var_name }} over the unscoped {{ var_name }} — it avoids collisions and makes templates self-documenting.

🧱
Use Blueprints for Rich Fields

Globals support blueprints too — use fieldtypes like assets, toggle, or bard for logo uploads, feature flags, or rich text content.

🚫
Don't Overload Globals

Globals are for site-wide data — not for content that varies by page. For page-specific content, use collections and entries instead.
