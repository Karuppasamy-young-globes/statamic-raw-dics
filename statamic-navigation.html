Introduction

🧭 What is Navigation?

A Nav in Statamic is how you build menus for your website — a structured hierarchy of links, entries, and text items that power your site's menus, like headers, footers, dropdowns, and sidebars. Each Nav is powered by a Structure, giving you a flexible, drag-and-drop interface to organise your menu items.

🏗️
Structure-Powered

Built on Statamic's Structure system — drag, drop, and nest items visually in the CP.

🔗
Flexible Links

Mix entries, custom URLs, and text-only items freely in a single nav.

📁
File-Based

Config and tree stored as YAML files — fully version-controllable, no database needed.

In simple terms

Nav = Structure + Links & Entries + Frontend Tag

Capabilities

⚡ What You Can Do

Navigation is more flexible than a simple list of links. Here's what a single Nav can handle:

🔗
Link to Entries

Pick entries directly from any of your collections. Their URLs always follow the collection's routing rules — no matter where they appear in the nav.

🌐
Add Custom URLs

Add internal paths or full external URLs — ideal for third-party links, marketing pages, or legacy URLs.

🏷️
Text-Only Items

Label-only items with no URL — great for dropdown section headings or non-clickable parent nodes.

🔁
Reuse the Same Entry

Reference the same entry multiple times across a single nav with no limits — useful for featured links or cross-section shortcuts.

👉 The CP gives you a drag-and-drop tree to build and nest your nav items visually.

⚡ Important

Entry URLs always follow their collection's routing rules — no matter where they appear in the nav. The nav never overrides URL generation.

On Disk

💾 Storage

Navigation data is split into two separate parts: the nav configuration and the navigation tree (structure). Each lives in its own directory.

Part 1 Nav Configuration

Defines settings like title, max depth, and which collections are selectable. One file per nav.

project structure
content/navigation/
  ├── header.yaml
  └── footer.yaml
content/navigation/footer.yaml
# content/navigation/footer.yaml

title: Footer
max_depth: 3
collections:
  - pages
  - posts
  - documents
Part 2 Navigation Tree (Structure)

Stores the actual hierarchy — the ordered "tree" of items. Edited visually in the CP; saved to YAML.

tree structure
content/trees/navigation/
  ├── header.yaml
  └── footer.yaml

Frontend

🧩 Templating

Rendering navigation in your frontend is simple using the nav tag in Antlers. The tag name matches the nav handle — e.g. nav:footer for the footer nav.

layout.antlers.html
<ul>
  {{ nav:footer }}
    <li>
      <a href="{{ url }}">{{ title }}</a>
    </li>
  {{ /nav:footer }}
</ul>
What You Get Inside the Loop
{{ title }}
The nav item's display label
{{ url }}
Resolved URL (entry route or custom URL)
{{ children }}
Nested items for dropdown rendering
Rendering Nested / Dropdown Menus
header.antlers.html (nested)
<ul>
  {{ nav:header }}
    <li>
      <a href="{{ url }}">{{ title }}</a>
      {{ if children }}
        <ul>
          {{ children }}
            <li><a href="{{ url }}">{{ title }}</a></li>
          {{ /children }}
        </ul>
      {{ /if }}
    </li>
  {{ /nav:header }}
</ul>
Entry Source

📦 Collections

By default, the entry selector in the CP shows content from all collections. You can restrict this in the nav config to keep things clean and relevant for editors.

Default (all collections)
# collections key omitted
# all collections appear in picker
Restricted to specific collections
footer.yaml
collections:
  - pages
  - blog
Editor UX

Restricting collections prevents editors from accidentally adding irrelevant content to a nav — e.g. blog posts appearing in the footer nav entry picker.

Custom Fields

🎨 Blueprints & Custom Data

Out of the box, nav items have two basic fields. Blueprints let you extend them with anything your design requires.

Default Fields
title url / entry
Example Custom Fields
excerpt icon badge css_class open_in_new_tab
🖥️ CP Integration

Custom fields defined in the nav blueprint appear automatically in the CP editor for each nav item — editors never need to touch YAML.

🔧 Any Field Type

Use text, toggle, asset, select, or any other Statamic fieldtype — the full power of blueprints is available.

Under the Hood

💾 How Data is Stored

Custom blueprint fields are saved under a data key in the tree YAML file, nested inside each nav item entry.

content/trees/navigation/header.yaml
-
  title: My page
  url: /my-page
  data:
    excerpt: This is my page
    icon: page.svg
title

The nav item label — can be different from the entry's title (useful for shortened menu labels).

url

For custom URL items. Entry-referenced items use the entry's route instead.

data

All custom blueprint fields live here — accessible in templates just like normal variables inside the nav loop.

Best Practices

💡 Developer Tips

🗂️
One Nav Per Zone

Create separate navs for separate areas — header, footer, sidebar. Avoid one "mega nav" that tries to do everything.

🎯
Restrict Collections per Nav

Lock down the collections list in each nav's config — editors will only see relevant entries in the picker, reducing mistakes.

🔭
Use max_depth Deliberately

Set max_depth: 1 for flat navs (footer) and higher values only when dropdowns are actually needed (header). Prevents accidental deep nesting.

🧩
Reuse Fields Across Blueprints

Define common fields (like icon) in a shared fieldset and import it into both entry and nav blueprints — consistent fieldtypes prevent override mismatches.
